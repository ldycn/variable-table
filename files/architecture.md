## 技术栈

- react
- typescript
- ant design
- tailwindcss
- Zustand
- playwright

## 按照下步骤，逐步完成需求

1. src目录下需要有apis,pages,ux,utils目录来完成对应功能，ux,utils参考/demo实现
2. 在src/apis中根据接口文档编写接口代码,fetch引用demo/fetch.ts
3. 在对应目录下严格按照下声明的界面目录名称和界面文件目录结构编写代码,要遵循下规则:
   1. 使用typescript进行类型标注
   2. 使用TailwindCSS编写样式
   3. 只要需求需要的组件是基础库文件夹中有的组件则不能使用ant design的组件实现对应功能
   4. 开发遵循SOLID原则,使用《设计模式：可复用面向对象软件的基础》中的设计模式。
   5. 表单项，表格组件全部使用基础库中的组件,不能使用ant design的组件
   6. 使用组件的时候，需要深入理解每一层调用组件的参数和意义并正确使用.
   7. 使用基础库表单组件的时候需要用form管理表单组件中的状态。相关组件的状态不要使用useState管理,在使用表单数据的地方使用form.getFieldsValue()完成功能
   8. 当请求出现错误时，将后端返回的msg信息通过message组件显示给用户
   9. 成功或者失败提示最后要加感叹号。
   10. 使用解构语法减少代码长度。
   11. 枚举必须在api文档中声明变量值和前端显示的名称,在界面中涉及到枚举值的地方直接使用对应api文档中的枚举值变量
4. 在./mock文件夹下生成api文档的mock数据,并默认在api文件中调用,调用方式为if(MOCK) { ... },MOCK变量声明在api文档全局变量中,默认使用MOCK数据
5. 检查界面是否能够正常运行,各个函数的入参类型否正确

执行过程中除了/demo目录下的文件外，不允许参考其他目录下的文件!

## 界面文件目录结构

除下文件之外，不允许添加其他文件!

```
page/
├── configs.ts # (必选) 界面配置声明 (表格列, 表单 Schema, 枚举等)
├── index.tsx # (必选) 页面 UI 结构 (仅负责组件渲染, 禁止业务逻辑/请求)
├── usePage.ts # (必选) 页面数据与业务逻辑 (核心文件)
├── modals/ # (可选) 弹窗相关代码 (所有弹窗逻辑集中在此)
└── components/ # (可选) 专用业务组件 (用于复杂页面的逻辑拆分)
```

### 文件/目录说明

#### configs.tsx

职责：存放静态配置，如 table 的列定义、表单项配置、重复组件,表单项声明或组件参数声明,页面内使用的枚举映射,权限字段等。

原则：保持纯数据和配置，无副作用。

#### index.tsx

职责：定义页面的最终 JSX 结构。

原则：纯粹的视图层。不允许包含数据获取、状态管理,事件处理,参数声明,重复组件,表单项声明或组件参数声明以及任何业务处理，仅负责渲染数据,声明界面结构和派发事件.

#### usePage.tsx

职责：处理所有数据逻辑、状态管理、API 请求、事件回调以及数据格式化。

原则：这是页面的核心逻辑层，负责连接视图与数据。

#### modals/

职责：存放该模块下使用的所有弹窗（Modal）组件。弹窗组件使用函数的方式调用。具体调用方法在src/ux/modal中查找

原则：当弹窗逻辑复杂或较多时，应将其拆分为独立的文件或组件放入此文件夹。

#### components/

职责：存放该页面专用的复杂子组件。

原则：用于拆分 index.tsx 重复，复杂，或独立的辑块，提高代码可维护性。
